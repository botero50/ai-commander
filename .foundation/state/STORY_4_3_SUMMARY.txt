================================================================================
KNOWN LIMITATIONS
================================================================================

1. Placeholder Agent Requirement
   - Framework requires at least one agent in WorldState
   - Workaround: Placeholder neutral agent created when no real units
   - Impact: Negligible (real games always have units)
   - Mitigation: Can be filtered with customData.isPlaceholder check

2. Map Position Count
   - One position per map tile (512×512 = 262,144 positions)
   - Memory: ~10-15MB per snapshot
   - Acceptable for typical 64-256 tile maps
   - Future: Can optimize with spatial indexing

3. Health-Only Resources
   - Only health mapped to resource pools
   - Cash/power stored in customData
   - Future expansion possible, not in scope

4. No Visibility Filtering
   - Observation shows full world state
   - By design: server-side sees all units
   - Framework supports client-side visibility filtering

================================================================================
ARCHITECTURE HIGHLIGHTS
================================================================================

Design Pattern: Adapter
  - OpenRA (external system) → Domain Model (internal)
  - Unidirectional dependency (OpenRA → AI Commander)
  - No coupling back to OpenRA

Immutability Strategy:
  - All WorldState objects frozen via Object.freeze()
  - Nested arrays/objects also frozen
  - Frozen customData prevents tampering

Determinism Guarantee:
  - Mapper is pure function (no side effects)
  - Same input always produces same output
  - Verified by tests on multiple runs

Caching Strategy:
  - Last 100 snapshots kept in memory
  - Historical queries supported via getWorldStateAt(tick)
  - Automatic pruning to prevent memory bloat
