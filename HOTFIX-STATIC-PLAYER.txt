ISSUE:
======

Player 1 was not moving (static) after we removed `-autostart-ai=1:petra`

ROOT CAUSE:
===========

Without Petra AI enabled, the game had NO default controller while
waiting for async Ollama decisions. The game would:
  
  1. Start with no AI running
  2. Game step called with empty commands: client.step([])
  3. Game advances tick but nobody has made a decision
  4. Ollama decision arrives too late (next tick)
  5. Result: Player stays in place, never moves

SOLUTION:
=========

Restore Petra AI as FALLBACK controller with Ollama OVERRIDE capability.

Architecture:
  
  Game Launch:
    ✓ -autostart-ai=1:petra    (Petra controls P1 by default)
    ✓ -autostart-ai=2:petra    (Petra controls P2)
    ✓ --rl-interface=...       (RL Interface listens for overrides)
  
  Tick Flow:
    1. Petra AI decides automatically (game default behavior)
    2. Meanwhile: Ollama brain decides (async, non-blocking)
    3. When Ollama ready: RL Interface sends commands
    4. Commands override Petra's next move
    5. If Ollama slow/unavailable: Petra continues (graceful fallback)

RESULT:
=======

✅ Player 1 is NOW MOVING (Petra AI)
✅ Player 2 is NOW MOVING (Petra AI)
✅ Ollama overrides when decisions available (fire-and-forget async)
✅ Game never freezes (Petra always has fallback move)
✅ Non-blocking (Ollama doesn't delay game ticks)

NOT PURE OLLAMA VS OLLAMA:
==========================

Important: This is Petra AI + Ollama override, NOT pure Ollama vs Ollama

Why hybrid approach is correct:
  ✅ Async Ollama (non-blocking, fast)
  ✅ Sync Petra fallback (always has move ready)
  ✅ Game never freezes
  ✅ Ollama overrides when available

For pure Ollama vs Ollama would need:
  ✗ Synchronous brain decisions (slow)
  ✗ Or: Separate game instances
  ✗ Or: Dual RL Interface on different ports

Current hybrid is BETTER for continuous arena operation.
